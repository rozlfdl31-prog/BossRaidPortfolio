#### 왜 그렇게 판단했는가

## Topic 4. Session 7 strict cleanup hardening
### 오늘 반영한 작업
* `MultiplayerSessionService`의 strict cleanup을 harden해서 `Closing` 상태를 real busy guard로 취급하고, fatal cleanup도 `_currentOperationTask` owner 안에서 같은 close path를 타도록 정리했다.
* lobby refresh / heartbeat / host `SessionState` update / optional lobby event callback에 session-version guard를 추가해, cleanup 이후 늦게 돌아온 async 결과가 새 session snapshot이나 in-flight flag를 덮지 못하게 막았다.
* `MultiplayerTitleSceneDriver`가 `StateChanged(Closing)`를 구독해 cleanup 시작 직후 추가 버튼 입력을 막고 `Closing...` label을 유지하도록 보강했다.

### 체크리스트 업데이트
* [x] `docs/roadmap/Milestone_Backlog.md`의 `2.7 strict cleanup 세션` 구현 완료
* [x] `dotnet build Assembly-CSharp.csproj -v:minimal` 컴파일 검사 통과
* [ ] Unity Editor Host + standalone Client strict cleanup manual smoke

### 맥락노트
* current runtime에는 basic strict cleanup path가 이미 있었고, 이번 범위는 그 owner를 바꾸기보다 race / stale async write / cleanup 중 재입력 같은 hardening에 집중했다.
* Session 7의 success bar는 "cleanup 중 새 action이 시작되지 않고, cleanup 완료 후에는 stale lobby/network state 없이 다시 title로 돌아간다"로 잡았다.
* `2P spawn`, `boss aggro`, `spectator`, `retry consensus`는 여전히 후속 세션 범위다.

### 기술적 고려
#### 무엇을 발견했는가
* callback 기반 fatal cleanup은 local UI 기준으로는 갑자기 시작되지만, service 쪽에서는 항상 `IsBusy`로 잡히지 않아 close 중 추가 `Create/Join/Start/Cancel` 시도가 겹칠 여지가 있었다.
* `RefreshLobbyAsync`, lobby heartbeat, host lobby metadata update 같은 async 작업은 cleanup 뒤에도 늦게 완료될 수 있어, 새 session이 시작된 뒤 old result가 snapshot/flag를 덮을 위험이 있었다.
* duplicated multiplayer title scene driver는 fatal error popup 시점에는 복구를 잘 했지만, cleanup 시작 직후 자체를 UI busy 상태로 반영하지는 않았다.

#### 무엇을 수정했는가
* service에 session-version counter를 추가하고 create/join/cleanup 시작마다 version을 올리도록 만들었다. tracked async path는 자신이 시작된 version이 current와 다르면 결과를 버린다.
* `BeginFatalShutdown()`은 idle tracked session에서는 real cleanup task owner가 되고, gameplay start 중 fatal case는 pending start fail로 접어 same cleanup path로 흘러가게 정리했다.
* title driver는 `Closing` state를 구독해 cleanup 우선순위를 UI에도 반영하고, fatal cleanup 후 label reset을 한 곳에서 처리하도록 다듬었다.

#### 왜 그렇게 판단했는가
* Session 7은 새 기능보다 "cleanup complete를 먼저 끝내고 그 뒤에만 새 action을 허용한다"는 계약을 닫는 단계라서, busy ownership과 stale async invalidation이 핵심이었다.
* cleanup 로직을 여러 UI branch에 흩뿌리기보다 `MultiplayerSessionService` 한 곳에서 authority를 유지해야 Lobby/Relay/NGO lifecycle 추적이 단순하다.
* session-version guard는 old callback을 개별 케이스로 다 막는 것보다 유지보수 비용이 낮고, 다음 multiplayer 세션에서도 같은 안전장치로 재사용할 수 있다.
