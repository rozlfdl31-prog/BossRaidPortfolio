## 맥락노트
* 현재 백로그에는 Attack1 공격범위를 `DamageCaster.radius` 대신 `BossController.basicAttackRange`로 수정하는 흐름이 적혀 있다.
* 오늘 작업 목표는 `basicAttackRange`를 Inspector 조정 포인트에서 내리고, 실제로 수정 가능한 공격범위 기준을 `DamageCaster.radius` 하나로 정리하는 것이다.
* 기존 씬 데이터(`GamePlayScene`, `GamePlayScene Scene_TestResult1`)를 확인했을 때 `basicAttackRange`와 `HeadDamageCaster._radius` 값이 이미 동일하게 저장돼 있어 source 전환 리스크는 낮았다.

## 구현 계획
### 목표
* Attack1의 사거리 판정과 실제 타격 반경이 모두 `HeadDamageCaster.radius`를 기준으로 동작하도록 바꾼다.
* `BossController.basicAttackRange`는 Inspector 조정 포인트에서 제거하고, Attack1 범위는 `DamageCaster` 쪽에서만 수정하게 만든다.

### 영향 파일
* `Assets/Scripts/Common/Combat/DamageCaster.cs`
* `Assets/Scripts/Boss/BossController.cs`
* `Assets/Scripts/Boss/BossFSM.cs` (필요 시 범위 참조 경로 정리)

### 변경 순서
* `DamageCaster`에 현재 반경을 읽을 수 있는 public read accessor를 추가한다.
* `BossController.BasicAttackRange`가 직렬화 필드가 아니라 `HeadDamageCaster.radius` 값을 반환하도록 바꾼다.
* `BossController`의 `basicAttackRange` 필드는 Inspector에서 숨기고, 기존 `SyncBasicAttackRangeToHeadDamageCaster()` 기반 동기화 경로는 제거한다. hidden field는 `HeadDamageCaster` 미할당 시 fallback 직렬화 값으로만 남긴다.
* Phase1의 Basic 패턴 선택, chase hysteresis의 최대 사거리 계산, gizmo 반경 표시가 모두 같은 `HeadDamageCaster.radius` 값을 읽도록 정리한다.
* `BasicAttackPattern`은 이미 `HeadDamageCaster.EnableHitbox(...)`를 사용하고 있으므로, 공격 실행 경로는 유지하고 range source만 단일화한다.

### 설계 근거
* `System_Blueprint`의 `Boss Basic Range-Hitbox Sync Rule`은 사거리 판정과 실제 타격 반경이 어긋나지 않도록 같은 값을 유지해야 한다고 정의하고 있다.
* 이번 변경은 sync direction을 `BossController -> DamageCaster`에서 `DamageCaster = single source of truth`로 바꾸는 작업이다.
* Inspector에서 같은 의미의 값을 두 군데 수정할 수 있으면 유지보수 중 혼선이 생기므로, 조정 포인트를 하나로 줄이는 편이 안전하다.

### 리스크
* 기존 프리팹/씬에서 `basicAttackRange`와 `HeadDamageCaster.radius`가 서로 다른 값이었다면, source of truth 변경 후 체감 사거리가 달라질 수 있다.
* `HeadDamageCaster` 참조가 누락된 프리팹에서는 Basic 사거리 계산이 비정상값이 될 수 있으므로 null fallback 또는 warning 처리 방안을 함께 검토해야 한다.
* gizmo와 실전 판정이 같은 값을 보도록 바꾸는 과정에서, 디버그 시각화가 임시로 어긋나지 않게 한 번에 정리해야 한다.

### 검증 포인트

* 현재 구현은 `readyDuration`을 별도 front delay가 아니라, 선택한 normalized slice의 재생 시간을 다시 맞추는 값으로 해석한다.
* 따라서 Attack1 전체 애니메이션을 멈추는 방식이 아니라, 지정한 bite 구간만 느리게/빠르게 재생되고 준비 구간이 끝난 뒤에만 실제 bite 판정이 열린다.
* Attack1 준비동작은 Attack2가 이미 사용 중인 MinMax slider UX를 그대로 재사용해 인스펙터 일관성을 맞췄다.

## 구현 계획
### 목표
* Attack1(물기)에 준비동작 구간을 추가하고, 디자이너가 인스펙터에서 준비 시간과 준비동작 normalized 구간을 직접 조절할 수 있게 만든다.
* 준비동작 동안에는 `HeadDamageCaster`가 비활성화된 상태를 유지하고, 준비가 끝난 뒤에만 실제 bite damage window가 열리게 만든다.

### 영향 파일
* `Assets/Scripts/Boss/BossController.cs`
* `Assets/Scripts/Boss/Attacks/BasicAttackPattern.cs`
* `Assets/Scripts/Boss/BossVisual.cs` (Attack1 playback helper가 필요할 경우)
* `Assets/Editor/EditorAssemblyAnchor.cs` (기존 MinMax drawer 재사용 여부 확인)

### 변경 순서
* `BossController`에 `BasicAttackSettings` 직렬화 데이터를 추가하고, `readyDuration`과 `readyNormalizedWindow`를 인스펙터에서 조절 가능하게 만들었다.
* `OnValidate()`에서 Attack1 준비 구간 값을 `0~1` 범위로 clamp하고, 종료값이 시작값보다 작아지지 않도록 보정했다.
* `BasicAttackPattern`을 단순 `attackDuration` 타이머 종료 구조에서, Animator `normalizedTime`을 읽는 구조로 바꿨다.
* 준비 구간 동안에는 hitbox를 열지 않고, 선택한 normalized window가 끝난 뒤에만 `HeadDamageCaster.EnableHitbox(...)`를 호출하도록 수정했다.
* `BossVisual`에 attack 전용 playback speed 제어 경로를 추가하고, 종료/인터럽트 시 기본값으로 복구하도록 구현했다.

### 설계 근거
* `System_Blueprint`는 패턴별 공격 타이밍을 인스펙터 값으로 조절 가능한 구조를 이미 사용 중이며, Attack2가 같은 방식의 `damageCastNormalizedWindow`를 사용하고 있다.
* Attack1 준비동작도 Controller가 설정 데이터를 들고, Pattern이 실행 타이밍을 해석하는 구조가 현재 Boss 공격 아키텍처와 가장 잘 맞는다.
* DamageCaster는 실제 판정 실행기이므로, 준비동작 동안 끄고 준비 종료 후에만 여는 방식이 요구사항과 가장 직접적으로 맞는다.

### 리스크
* Attack1 playback speed를 조절할 경우, 종료 시 speed 복구가 누락되면 이후 Locomotion이나 다른 공격 애니메이션에 영향이 갈 수 있다.
* CrossFade 구간이나 상태명 불일치가 있으면 `normalizedTime` 기반 준비 구간 판정이 늦게 시작되거나 누락될 수 있다.
* `readyDuration = 0` 또는 매우 짧은 값에서 현재 bite와 거의 같은 체감이 나와야 하므로 edge case 처리가 필요하다.
* 피격/사망/강제 전환 시 Attack1 hitbox가 남아 있지 않도록 exit 정리가 확실해야 한다.

### 검증 포인트
* `readyDuration = 0`일 때 현재 Attack1과 거의 같은 체감으로 동작해야 한다.
* `readyNormalizedWindow` 슬라이더를 옮기면 준비동작으로 취급되는 bite 구간이 함께 바뀌어야 한다.
* 준비동작 동안 `HeadDamageCaster`가 열리지 않고, 준비 종료 후에만 damage가 들어가야 한다.
* Attack1 도중 피격/사망/상태 전환이 발생해도 hitbox와 playback 상태가 정상 복구되어야 한다.
* 코드 구현 후 `dotnet build Assembly-CSharp.csproj -v:minimal`로 컴파일 검증을 수행해야 한다.

## 기술적 고려
### 무엇을 발견했는가
* 현재 live `BasicAttackPattern`은 `Enter()`에서 bite 애니메이션 재생과 동시에 `HeadDamageCaster.EnableHitbox(...)`를 호출하므로 Attack1에는 준비동작이 없다.
* Attack2는 이미 `MinMaxRangeAttribute` 기반 `Vector2 damageCastNormalizedWindow`를 사용하고 있어, Attack1도 같은 인스펙터 UX를 재사용할 수 있다.
* `BossVisual`은 기존에 locomotion speed 파라미터 제어만 있었고, Attack1 준비 구간을 늘이기 위한 `Animator.speed` 복구 helper는 없었다.
* 첫 빌드에서 `BossController.BasicAttackSettings` nested type과 같은 이름의 property가 충돌해 컴파일 에러가 발생했고, accessor 이름을 `BasicAttackConfig`로 바꾸며 해결했다.

### 무엇을 수정했는가
* `BossController`에 `BasicAttackSettings`와 clamp 로직, `BasicAttackConfig` accessor를 추가했다.
* `BasicAttackPattern`을 normalizedTime 기반으로 바꾸고, 준비 구간 동안 hitbox를 닫은 채 유지하다가 준비 종료 후 damage window를 여는 흐름으로 수정했다.
* `BossVisual`에 `SetAnimatorPlaybackSpeed`, `ResetAnimatorPlaybackSpeed`, `GetBasicAttackClipLengthOrDefault` helper를 추가해 Attack1 ready slice 재생 시간을 직접 제어하게 했다.
* `Progress_Log`, `System_Blueprint`, `Technical_Glossary`, `Milestone_Backlog`를 구현 결과 기준으로 동기화했다.

### 왜 그렇게 판단했는가
* 사용자 요구사항의 핵심은 "how long it readies"와 "which part of the bite animation is ready motion"을 인스펙터에서 직접 조절하는 것이므로, 단순 고정 딜레이보다 normalized window 기반 설계가 더 잘 맞는다.
* 현재 Boss 공격 구조는 `Controller = config holder`, `Pattern = timing executor` 역할 분리가 분명하므로, Attack1도 같은 구조를 유지하는 편이 유지보수와 문서 일관성에 유리하다.
* 준비동작 동안 damage caster를 끄고, 종료/인터럽트 시 `Animator.speed`와 hitbox를 모두 정리해야 회귀 리스크를 가장 작게 만들 수 있다.
