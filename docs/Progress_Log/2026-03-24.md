### 기술적 고려
#### 무엇을 발견했는가
* 기존 shared repo는 서드파티 imported pack 상당수를 Git LFS로 직접 배포하고 있었고, 무료 bandwidth 한도에 닿으면 teammate pull이 바로 막혔다.
* vendor pack을 Git에서 제거하더라도 `.meta`까지 같이 Drive에 보관하지 않으면 Unity GUID reference가 쉽게 깨질 수 있다.
* `git rm --cached`를 사용하면 local file은 그대로 두고 repository tracking만 끊을 수 있어, cutover 당일 작업 손실 위험이 가장 낮았다.

#### 무엇을 수정했는가
* Google Drive 외부 배포 대상 pack 경로와 pack root `.meta`를 `.gitignore`에 명시했다.
* 기존 vendor-pack Git LFS 규칙을 `.gitattributes`에서 제거했다.
* 대상 pack 전체를 index에서 제거하고, 문서를 새 외부 배포 정책에 맞게 갱신했다.

#### 왜 그렇게 판단했는가
* 이번 이슈의 본질이 LFS bandwidth였으므로, vendor pack을 계속 Git/LFS로 보관하는 한 teammate pull risk를 근본적으로 줄일 수 없다.
* pack별 selective mix 대신 "모든 third-party imported pack은 Drive" 규칙이 팀 운영상 더 단순하고, 누구나 같은 restore step을 따르면 된다.
* cutover commit 이후에는 Git clone/pull이 가벼워지고, asset restore 책임은 `Assets.zip` 같은 Drive artifact로 분리되어 재현성이 높아진다.
