# 2026-03-06 (금): Boss Attack4 AoE 반경 판정 규칙 수정

## 오늘 반영한 작업
* Attack4 AoE 데미지 타이밍을 fully red(active) 구간 기준으로 정리했다.
* `AoECircleController`의 active phase 틱 루프를 제거하고, fully red 동안 반경 내 대상을 프레임마다 검사하되 circle당 1회만 적용하도록 변경했다.
* `EnterActivePhase()` 진입 시 즉시 1회 반경 판정을 수행해 fully red 시작 프레임 타이밍 오차를 줄였다.
* `ReceiveBossAttackHit` 결과가 `Ignored`(invul)인 경우 hit consumed 처리하지 않고, fully red 구간 내에서는 invul 종료 후 1회 피격 가능하게 유지했다.
* AoE 데미지 소스는 기존 `AoEAttackSettings.damage`를 유지했다.

## 체크리스트 업데이트
* [x] Attack4 AoE fully red 구간 반경 판정 적용
* [x] one circle, one hit 규칙 적용
* [x] fully red 시작 프레임 즉시 판정 적용
* [x] Progress_Log/Blueprint/Glossary 문서 동기화

## 맥락노트
* 요청 스펙의 핵심은 `one-time hit`와 `fully red 중 later entry도 피격`의 동시 만족이다.
* 구현 정책은 "active window one-hit per target per circle"로 고정했다.
* Attack4는 기존 `BossAttackHitType.Attack4Projectile` 경로를 유지해 플레이어 stun/post-stun invul 처리와 동일 파이프라인을 사용한다.

## 기술적 고려
### 무엇을 발견했는가
* 기존 AoE는 active phase에서 `tickInterval` 기준 반복 데미지를 적용했고, tick마다 hit set을 초기화해 same circle 재피격 가능성이 있었다.
* telegraph 종료 프레임과 첫 데미지 프레임 사이에 1프레임 이상 지연이 발생할 수 있었다.

### 무엇을 수정했는가
* active phase 반복 틱 루프를 제거하고, fully red 동안 프레임 단위 반경 검사로 전환했다.
* circle 생명주기 단위 hit set 유지로 same circle 재피격을 차단했다.
* `EnterActivePhase()`에서 즉시 1회 판정을 추가했다.
* `AoEAttackPattern`의 `StartTelegraph` 호출 시그니처를 업데이트했다(틱 인자 제거).

### 왜 그렇게 판단했는가
* 요구사항은 projectile 충돌이 아니라 AoE radius + fully red 기준 판정이다.
* later entry 피격과 one circle one hit을 동시에 만족하려면 active window 동안 신규 대상만 1회 적용하는 방식이 가장 직접적이다.
* invul 처리 일관성을 위해 기존 `BossAttackHitResolution` 경로를 그대로 재사용하는 것이 회귀 위험이 가장 낮다.

## 추가 반영 (용어 정리: telegraph -> warning)

### 오늘 반영한 작업
* Attack3/Attack4 관련 코드에서 `telegraph` 용어를 `warning`으로 정리했다.
* 런타임 동작은 유지하고, 직렬화 필드에는 `FormerlySerializedAs`를 추가해 기존 씬/프리팹 값 호환성을 유지했다.
* `AoECircleController` 공개 API를 `StartTelegraph`에서 `StartWarning`으로 변경하고 호출부를 동기화했다.

### 체크리스트 업데이트
* [x] Attack3/Attack4 설정 용어를 `warningDuration`으로 통일
* [x] AoE circle API 용어를 `StartWarning`으로 통일
* [x] 직렬화 호환성(`FormerlySerializedAs`) 유지
* [x] Progress_Log/Blueprint/Glossary 문서 동기화

### 맥락노트
* 사용자가 `telegraph` 단어를 혼동했기 때문에, 같은 의미의 쉬운 단어(`warning`)로 코드 읽기 난이도를 낮추는 것이 우선이었다.
* 런타임 리스크를 줄이기 위해 값 의미와 타이밍 로직은 변경하지 않고 이름만 정리했다.

### 기술적 고려
#### 무엇을 발견했는가
* `telegraphDuration`과 `StartTelegraph` 같은 이름은 신규 개발자에게 의미 전달이 직관적이지 않았다.

#### 무엇을 수정했는가
* `telegraphDuration` -> `warningDuration`
* `StartTelegraph` -> `StartWarning`
* `_telegraphTimer`/`_telegraphDuration`/`telegraphRenderer`/`telegraphColor` -> `warning*` 계열

#### 왜 그렇게 판단했는가
* 변경 목표는 "동작 변경"이 아니라 "읽기 쉬운 의미 전달"이다.
* 이름 정리 + 직렬화 호환을 같이 적용하면 유지보수성과 안정성을 동시에 확보할 수 있다.

#### 무엇을 수정했는가

#### 왜 그렇게 판단했는가

* 인스펙터에서 "warningDuration 하나만 조정하면 된다"는 규칙이 명확해진다.
