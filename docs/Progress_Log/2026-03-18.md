## 콤보 UI 뜨게 하기

### 추가 요청 대응 계획 (boss hit confirm only)
* 문제 원인은 `AttackState.StartComboStep()`가 공격 step 시작 시점에 바로 combo UI를 열고 있어, 실제 boss hit 여부와 상관없이 UI가 보인다는 점으로 정리했다.
* 수정은 combo UI의 show timing만 `DamageCaster`의 real hit confirm 경로로 옮기고, hide timing(`AttackState.Exit()`, death path, HUD init, `ShowHud(false)`)은 유지하는 방향으로 진행한다.
* `DamageCaster`에는 `didDamage == true`일 때만 1회 발행되는 hit-confirm event를 추가하고, `PlayerController`가 이 event를 받아 현재 pending combo step을 HUD에 표시하게 만든다.
* `AttackState`는 combo step 숫자를 직접 보여주지 않고, 현재 단계만 `PlayerController`에 pending 값으로 넘기도록 바꾼다.
* 검증 기준은 `miss -> hidden 유지`, `boss hit -> combo UI 표시`, `next hit -> 숫자 갱신`, `dash cancel/hit/stun/death/end -> hide`다.

### 오늘 반영한 작업
* `Assets/Scripts/Common/Combat/DamageCaster.cs`에 `OnAttackHitConfirmed` event를 추가했다. 같은 공격 윈도우 안에서는 첫 번째 `didDamage == true` 순간에만 1회 발행되도록 막았다.
* `Assets/Scripts/Player/PlayerController.cs`가 combo step을 `pending` 상태로 들고 있다가, `DamageCaster.OnAttackHitConfirmed`를 받는 순간에만 `ShowComboHud(step)`를 호출하도록 바꿨다.
* `Assets/Scripts/Player/States/AttackState.cs`는 공격 step 시작 시 combo UI를 직접 열지 않고, 현재 step 숫자만 `SetPendingComboHudStep(step)`로 준비하도록 수정했다.
* 기존 hide 경로(`AttackState.Exit()`, death path, HUD init, `ShowHud(false)`)는 유지해 stale combo UI 정리 동작은 그대로 보존했다.
* `Canvas.prefab` 안의 existing `Text_Combo` / `Image_Combo` 구조를 그대로 재사용해, scene hierarchy 재구성 없이 code gate만으로 visibility를 처리했다.

### 목표
* `Session 5 - Lobby Active Session`을 runtime에서 실제 동작하게 만든다.
* Host와 Client가 같은 `LobbyPanel`에서 real room state를 본다.
* Host `Start` 버튼은 `2/2 connected + NGO stable + 2초 유지`일 때만 활성화되게 만든다.
* 이번 범위는 lobby active까지만 다루고, `Session 6 - gameplay start`의 NGO synchronized scene load는 다음 단계로 남긴다.

### 영향 파일
* `Assets/Scripts/Multiplayer/Services/MultiplayerSessionService.cs`
* `Assets/Scripts/Multiplayer/UI/MultiplayerTitleSceneDriver.cs`
* 필요 시 표시 bridge만 확인: `Assets/Scripts/Common/TitleSceneController.cs`

### 설계 메모
* Session owner는 계속 `MultiplayerSessionService`로 유지한다. UI는 Relay/Lobby/NGO 호출을 직접 하지 않고 snapshot 값만 받아서 표시한다.
* Host는 `15s` heartbeat를 유지하고, lobby metadata의 `SessionState`를 `Waiting` / `Ready`로 관리한다.
* Lobby state 반영은 `Lobby Events 우선 + Poll fallback` 구조를 사용한다. Wire package가 없을 때도 poll만으로 state convergence가 되도록 만든다.
* `CanStart` 계산은 service 내부에서만 수행한다. UI는 `IsHost`, `ConnectedPlayerCount`, `LobbyStatusText`, `CanStart`만 바인딩한다.
* no-op prototype countdown과 분리해서, real runtime countdown은 NGO connected state와 lobby player count를 같이 보고 판단한다.

### 구현 단계
1. `MultiplayerSessionService`에 lobby-active poll timer를 추가하고, `LobbyActive` 동안 주기적으로 `GetLobbyAsync` refresh를 수행한다.
2. snapshot builder를 확장해 `ConnectedPlayerCount`, `LobbyStatusText`, `CanStart`를 real session state 기준으로 계산한다.
3. Host 기준 `2/2 + NGO stable + 2초 유지` 타이머를 service에 넣고, 안정 구간이 깨지면 즉시 timer를 리셋한다.
4. Host가 ready/unready 변화를 감지하면 lobby metadata의 `SessionState`를 갱신해 client UI도 같은 status text를 받게 만든다.
5. client leave / disconnect / host cancel / lobby gone 상황에서 Session 5가 `Waiting` 복귀 또는 strict cleanup으로 올바르게 나뉘는지 정리한다.
6. `MultiplayerTitleSceneDriver`는 local countdown을 만들지 않고 service snapshot만 그대로 표시하도록 유지한다.

### 리스크
* poll 주기가 너무 짧으면 Lobby API 호출이 불필요하게 많아질 수 있다.
* host-side ready 판단과 client-side snapshot 갱신 타이밍이 엇갈리면 `Start` 표시가 순간적으로 흔들릴 수 있다.
* heartbeat, refresh, cleanup이 겹치는 프레임에서 중복 shutdown 또는 stale snapshot publish가 발생할 수 있다.

### 검증 시나리오
* Host create 후 `LobbyPanel`에 real `1/2 connected`와 room data가 유지된다.
* Client join 후 Host/Client 모두 real `2/2 connected`를 본다.
* Host만 `Start` 버튼을 보고, `2초 stable` 전에는 비활성 상태를 유지한다.
* `2초 stable`이 지나면 Host `Start`가 활성화되고, client는 계속 비활성/hidden 상태를 유지한다.
* Client가 나가면 Host UI가 다시 `1/2 waiting`으로 돌아간다.
* Host cancel 또는 lobby delete 시 client도 cleanup 후 title로 돌아간다.

### 오늘 반영한 작업 (Lobby active 세션)
* `Assets/Scripts/Multiplayer/Services/MultiplayerSessionService.cs`에 `LobbyActive` 전용 poll fallback 타이머를 추가했다. Wire package가 없는 기본 경로에서도 `2s` 주기로 lobby refresh를 다시 시도한다.
* Host `Start` unlock 규칙을 real runtime으로 옮겼다. `2/2 connected + NGO stable + 2초 유지`를 만족할 때만 `CanStart`가 `true`가 된다.
* Host가 lobby metadata의 `SessionState`를 `Waiting` / `Ready`로 직접 갱신하도록 만들었다. client UI는 이 값을 poll/event refresh로 받아 same lobby status를 표시한다.
* snapshot builder가 더 이상 `canStart = false`를 고정하지 않고, real `ConnectedPlayerCount`, `LobbyStatusText`, `CanStart`를 계산해 `LobbyPanel`에 전달한다.
* Host는 플레이어 수 판단에서 Lobby count의 `max`가 아니라 NGO 연결 상태를 우선 source로 사용하도록 바꿨다. 그래서 client leave 직후 host UI가 즉시 `1/2 waiting`으로 돌아간다.
* client local NGO disconnect, lobby gone, membership loss 같은 Session 5 실패 케이스를 fatal cleanup으로 연결했다.

### 맥락노트
* default package 구성에서는 Lobby Events가 optional path라서, no-Wire 기본 경로에서도 state convergence가 되는 poll fallback이 필요했다.
* 기존 `ResolveConnectedPlayerCount()`는 host에서 `max(lobby, network)`를 써서 client join에는 유리했지만, client leave 직후 `1/2` 복귀가 느려지는 문제가 있었다.
* client는 host의 local NGO stable 상태를 직접 알 수 없으므로, `Ready` 여부를 host-owned lobby metadata로 내려주는 구조가 가장 단순했다.
* Session 6 synchronized gameplay start는 아직 별도 범위이므로, 이번 단계에서는 `LobbyPanel` active state와 `Start` unlock까지만 닫았다.

### 기술적 고려
#### 무엇을 발견했는가
* `MultiplayerSessionService.BuildCurrentSnapshot()`은 Session 5 이전까지 `CanStart`를 항상 `false`로 고정하고 있었다.
* no-Wire 기본 경로에서는 lobby state가 network callback이나 수동 refresh에만 의존해, long-wait 상황에서 latest lobby state 반영이 불안정했다.
* host 기준 player count 계산은 join 직후에는 빨랐지만, disconnect 직후에는 stale lobby player count 때문에 `1/2 waiting` 복귀가 늦을 수 있었다.
* client local NGO disconnect와 `GetLobbyAsync` 기반 lobby gone/membership loss는 Session 5 failure mapping이 더 필요했다.

#### 무엇을 수정했는가
* `MultiplayerSessionService`에 `LobbyActive` 전용 poll fallback, host start stable timer, host-owned `SessionState` sync, snapshot dedupe publish, disconnect/lobby gone cleanup 분기를 추가했다.
* host는 NGO connected client count를 우선 source로 사용해 `ConnectedPlayerCount`를 계산하도록 조정했다.
* `LobbyStatusText`는 host/local stable gate와 client/lobby metadata를 조합해 `waiting`, `waiting for host`, `ready` 상태를 구분하도록 정리했다.
* `docs/roadmap/Milestone_Backlog.md`, `docs/technical/System_Blueprint.md`, `docs/technical/Technical_Glossary.md`, `docs/Progress_Log/README.md`를 `2026-03-18` 로그 기준으로 갱신했다.

#### 왜 그렇게 판단했는가
* Session owner를 계속 `MultiplayerSessionService` 하나로 유지해야 UI가 service state만 바인딩하고, 이후 Session 6 scene sync를 붙일 때 변경 범위를 작게 가져갈 수 있다.
* Host readiness는 Host만 정확히 판단할 수 있으므로, client가 직접 추론하기보다 host가 `SessionState` metadata를 써 주는 쪽이 안전하다.
* countdown 숫자를 UI 프로토타입처럼 local fake timer로 다시 만들기보다, service가 `CanStart`와 status text를 직접 계산하는 구조가 runtime truth source에 더 가깝다.
* disconnect와 lobby gone을 cleanup owner 한 곳에서 닫아야 Session 5와 Session 7 사이의 경계가 단순해지고 stale session 위험이 줄어든다.
