# 2026-03-04 (수): (보스) Attack2/Lunge 실험 전체 정리 및 실패 원인 확정

## 오늘 반영한 작업
* 어제~오늘 진행한 Attack2/Lunge 실험 항목을 누락 없이 복구 정리했다.
* 실험 기록을 `시도 -> 결과 -> 실패 원인` 형식으로 통일했다.
* 실패 실험(Soft Lock/Hard Lock/일부 타이밍 실험)은 "시도 이력"으로 유지하고, 현재 런타임 코드에서 제거/롤백된 상태를 명시했다.
* 현재 기준선(실험 9 반영 코드) 기준으로 아래 항목을 유지 중이다.
  * `ApplyLungeRootMotion(Vector3 deltaPosition, float normalizedTime = -1f)` 경로
  * `LungeAttackSettings`의 `midBoost*`/`lateReduce*` 이동량 분포 보정
  * RootMotion 로그 `scale` 추적 필드

## 실험별 기록 (시도 -> 결과 -> 실패 원인)

### 1) 토글 3종 분리 실험
* 시도:
  * `useAnimatorDeltaOnlySource`
  * `useDirectRootMotionDelta`
  * `suppressGravityWhileLungeRootMotion`
  * `ShowPhaseDebugeLabel` 조합 A/B 테스트
* 결과:
  * `useAnimatorDeltaOnlySource` 단독: 체감 변화 거의 없음
  * `useDirectRootMotionDelta` 포함: 슬라이드 재현
  * `suppressGravityWhileLungeRootMotion` 단독: 체감 변화 거의 없음
  * `useDirectRootMotionDelta + suppressGravityWhileLungeRootMotion`: 슬라이드 + 부유 증가
* 실패 원인:
  * 토글 조합 편차는 컸지만 Red-Boss 동기화는 일관 개선되지 않았다.
  * `Move(deltaPosition)` 직결 경로에서 과거 슬라이드 체감이 재현됐다.

### 2) Anchor 동기화 실험
* 시도:
  * `useAnchorDeltaSync = checked + red`
  * `useAnchorDeltaSync = checked + boss`
  * `useAnchorDeltaSync = unchecked (+ red/boss)` 비교
* 결과:
  * `checked + red`: red 점프, boss 제자리
  * `checked + boss`: 자기 참조로 이동 0 유지
  * `unchecked`: 기존 관통 점프는 유지되나 동기화 실패
* 실패 원인:
  * anchor 경로에서 유효 이동 입력이 0으로 반복되는 구간이 발생했다.
  * anchor 선택만으로 기준점 오차를 안정적으로 줄이지 못했다.

### 3) Soft Lock Sync 1차
* 시도:
  * `useSoftLockSync` ON, `gain/maxCorrection` 설정, 런지 1회 로그 수집
  * 로그 파일: `docs/console_debug_log.txt`
* 결과:
  * `softLock(on=True)`인데 `used=False`, `errorXZ=0.000`, `corrXZ=0.000` 반복
  * `baseDeltaXZMag`와 `appliedDeltaXZMag` 동일 패턴 유지
  * Exit 로그: `deltaXZ=(0.445,3.657)` 오차 잔존
* 실패 원인:
  * 보정항 입력이 0으로 계산되어 `correctionDelta`가 생성되지 않았다.
  * 결과적으로 기존 `CharacterController.Move(deltaPosition)` 경로와 차이가 없어 개선 불가.

### 4) 기준선 복구 롤백
* 시도:
  * Soft Lock/Anchor/실험 분기 롤백
* 결과:
  * 비교 가능한 기준선은 복구됨
* 실패 원인:
  * 문제 해결이 아니라 실험 누적으로 흐려진 기준 복구 단계였다.

### 5) Hard Lock Sync 1차 구현
* 시도:
  * Lunge 시작 시 `initialOffset = boss - red` 저장
  * 런지 중 매 프레임 `target = red + initialOffset`, `move = target - current` 적용
  * `hardLockMaxStep` 클램프 추가
* 결과:
  * 구현 및 컴파일은 성공
* 실패 원인:
  * 실플레이 검증 전에는 성공/실패 판정 불가

### 6) Hard Lock 실플레이 3회 검증
* 시도:
  * `useHardLockSync=ON, hardLockMaxStep=0.6`
  * `useHardLockSync=ON, hardLockMaxStep=0.4`
  * `useHardLockSync=OFF, hardLockMaxStep=0.4`
* 결과:
  * Hard Lock ON(0.6/0.4): 보스가 전방이 아니라 측면 이탈
  * Hard Lock OFF: 전방 도약 정상
* 실패 원인:
  * `maxStep` 문제가 아니라 Hard Lock 목표 벡터 구성(기준점 해석) 문제였다.
  * 이동 API 자체보다 입력 벡터가 잘못 계산된 것이 직접 원인.

### 7) Hard Lock/실험 잔해 코드 정리
* 시도:
  * `BossController`/`BossVisual`/`LungeAttackPattern`에서 Hard Lock 전용 경로 제거
  * 테스트 씬 직렬화 잔해 정리
* 결과:
  * 실험 분기 오염 제거, 기준선 단순화
* 실패 원인:
  * 해결 단계가 아닌 "검증 가능한 상태 복구" 단계
* 현재 상태:
  * Hard Lock/Soft Lock 런타임 경로는 기준선에서 제거 유지

### 8) 애니메이션 타이밍 데이터화 시도(중간 실험)
* 시도:
  * Lunge 타이밍/게이트를 데이터화하는 코드 실험 진행
  * 로그 기반 비교로 전이/적용 구간 검증
* 결과:
  * 타이밍 제어 자체는 동작했지만 동기화 복원에는 유효하지 않았음
* 실패 원인:
  * "언제 적용할지"만 바꿨고, "어디로 이동할지"(기준점/방향) 문제는 남아 있었음
* 현재 상태:
  * 해당 데이터화 실험 코드는 기준선에서 롤백/정리됨

### 9) Attack2 55프레임 이동 분포 보정 (실험 9)
* 시도:
  * `midBoost/lateReduce` 배수로 구간별 이동량 분포 보정
  * `ApplyLungeRootMotion(..., normalizedTime)` 경로에서 배수 적용
  * RootMotion 로그에 `scale` 출력 추가
* 결과:
  * `scale != 1` 로그가 반복되어 배수 적용 자체는 확인됨
  * 중간 구간 정체 일부 완화 시도는 있었으나, 동기화 복원은 확인되지 않음
  * Exit 오차 축은 여전히 크게 남음
* 실패 원인:
  * 이번 보정은 이동 `크기` 조절이며, Red-Boss 기준점 오차를 직접 보정하지 못함
  * 이동 경로가 방향 잠금 + `CharacterController.Move(...)` 기반이라 동기화 핵심축은 잔존
* 현재 상태:
  * 실험 9 배수 파라미터/로그는 현재 코드에 유지되어 비교 실험 축으로 사용 중

## 맥락노트
* 핵심은 "무슨 실험이 왜 실패했는지"를 누락 없이 보존하는 것이다.
* 실패 실험을 지우면 재발 시 비교 기준이 사라진다.
* 그래서 현재 코드 반영 여부와 무관하게, Attack2 시도 이력 자체를 실험 기록으로 유지한다.

## 기술적 고려
### 무엇을 발견했는가
* Attack2/Lunge 이슈는 단일 파라미터 튜닝 문제가 아니라, 전이/기준점/이동벡터 해석이 얽힌 복합 문제다.
* 전이 안정화(마커/타이밍)와 동기화 복원(기준점/벡터)은 분리해서 봐야 한다.

### 무엇을 수정했는가
* 누락된 실험 이력(1~9)을 재복구하고, 각 시도의 결과/원인을 명시했다.
* 현재 기준선에 남아있는 실험 9 반영 상태와 롤백된 과거 실험을 구분해 기록했다.

### 왜 그렇게 판단했는가
* 실패 기록이 누락되면 같은 실험을 반복하고 비교 비용이 커진다.
* 포트폴리오/블로그 목적에서도 "실패 원인 분리 능력"이 핵심이므로, 시도-결과-원인 형태로 보존하는 게 맞다.
