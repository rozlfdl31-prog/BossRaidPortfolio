# 🏗️ 디자인 패턴 비교 분석: FSM vs Strategy (그리고 숨겨진 패턴들)

> **"비슷해 보이는 FSM과 전략 패턴, 도대체 무슨 차이인가?"**
> 
> 보스 레이드 프로젝트를 진행하며 `StateMachine`과 `Strategy Pattern`을 모두 사용했다. 구조적으로 유사해 보이는 두 패턴의 결정적 차이와, 프로젝트 곳곳에 숨겨진 다른 패턴들을 분석해 본다.

---

## 1. FSM (Finite State Machine) vs Strategy Pattern

두 패턴 모두 **'인터페이스(또는 부모 클래스)를 통한 다형성'**을 핵심으로 하기 때문에 클래스 다이어그램만 보면 매우 똑같다. 하지만 그 **목적**과 **주체**에서 명확한 차이가 있다.

### 1.1. 결정적 차이점 비교

| 특징 | 상태 패턴 (FSM) | 전략 패턴 (Strategy) |
| :--- | :--- | :--- |
| **핵심 질문** | **"나는 누구인가?"** (Who am I?) | **"어떻게 할 것인가?"** (How do I do?) |
| **목적** | 객체의 **내부 상태**를 정의하고 행동을 변경 | 특정 **알고리즘(행동)**을 캡슐화하여 교체 |
| **주체** | **플레이어/보스 전체**가 상태를 가짐 | **특정 상태(State)**가 전략을 가짐 |
| **전환** | 행동이 상태 자체를 바꿈 (`Idle` → `Attack`) | 상태 안에서 행동 방식만 바꿈 (`Attack A` ↔ `Attack B`) |
| **비유** | 사람이 **'잠자는 상태'**에서 **'달리는 상태'**로 변함 | 달리는 사람이 **'나이키'**를 신었다가 **'아디다스'**로 갈아 신음 |

### 1.2. 프로젝트 적용 사례

*   **FSM (`BossController`)**:
    *   보스는 한 번에 하나의 상태(`Idle`, `Attack`, `Dead`)만 가진다.
    *   `Idle`에서 플레이어를 발견하면 `Move`로, 체력이 다하면 `Dead`로 **상태 자체가 전이**된다.
*   **Strategy (`BossAttackState`)**:
    *   보스가 `Attack` 상태일 때, 그 내부에서 **'어떻게 공격할지'**를 결정한다.
    *   `StartAttack()`을 호출하면 상황(거리, 쿨타임)에 따라 `BasicAttackPattern` 또는 `SkillAttackPattern` 전략을 갈아끼워 실행한다.

## 2. 프로젝트에 녹아있는 다른 디자인 패턴들

단순히 FSM과 전략 패턴만 쓰인 것이 아니다. Unity 프로젝트 특성상 자연스럽게 사용된 패턴들을 재정의해 보았다.

### 2.1. 템플릿 메서드 패턴 (Template Method Pattern)
> *"부모가 큰 틀을 잡고, 자식이 세부 내용을 채운다."*

*   **적용 위치**: `BaseState<T>`와 그 자식들 (`MoveState`, `AttackState`)
*   **분석**: `StateMachine`은 `IState.Enter()`만 호출하면 된다. 그 내부에서 애니메이션을 재생할지, 타이머를 초기화할지는 자식 클래스가 결정한다. 전체 흐름(Enter -> Update -> Exit)의 골격은 부모가 정의하고, 구체적인 내용은 자식이 채우는 구조다.

### 2.2. 커맨드 패턴 (Command Pattern - Data Driven)
> *"명령을 데이터로 포장해서 보낸다."*

*   **적용 위치**: `PlayerInputPacket`
*   **분석**: `Update()`에서 `if (Input.GetKeyDown...)`으로 직접 입력을 확인하지 않고, 입력 자체를 `InputPacket`이라는 **데이터 객체로 캡슐화**하여 전달한다. 이는 향후 네트워크 패킷으로 그대로 전송할 수 있는 유연성을 제공한다. (엄밀히 말하면 커맨드 패턴의 데이터화 응용이다.)

### 2.3. 컴포넌트 패턴 (Component Pattern)
> *"기능을 부품처럼 조립한다."*

*   **적용 위치**: `BossVisual`, `DamageCaster`, `Health`
*   **분석**: `BossController`라는 거대한 클래스가 모든 일을 하지 않는다.
    *   **시각적 처리**: `BossVisual` 부품이 담당
    *   **공격 판정**: `DamageCaster` 부품이 담당
    *   **체력 관리**: `Health` 부품이 담당
    이 부품들을 `BossController`가 조립(Composit)하여 하나의 온전한 보스로 동작하게 만드는 것이 컴포넌트 패턴의 핵심이다.

## 3. 결론 (면접 대비 요약)

> "이 프로젝트는 캐릭터의 생명주기 관리를 위해 **FSM**을 사용했고, 다양한 보스 공격의 유연한 확장을 위해 **전략 패턴**을 적용했다.
> 
> 또한, 코드 재사용성을 높이기 위해 **템플릿 메서드 패턴**(`BaseState`)을 활용했으며, 입력 시스템은 향후 네트워크 확장을 고려하여 **커맨드 패턴**의 아이디어를 차용한 패킷 구조로 설계했다."

이처럼 단순한 구현을 넘어, 각 패턴의 **존재 이유(Why)**를 명확히 이해하고 적재적소에 배치하는 것이 중요하다.
